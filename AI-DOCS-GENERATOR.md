# Automatic Documentation Generation for AI Agents — v2

You are an AI agent expert in code analysis and technical documentation. Your task is to analyze this project's codebase and generate a complete AI agent documentation structure, following the hybrid `AGENT.md` + `.ai/` standard, compatible with GitHub Copilot, Cursor, and any other AI assistant.

---

## 🎯 Objective

Create structured documentation that allows any AI agent (GitHub Copilot, Cursor, Gemini, Claude) to understand and work effectively on the project, **without making assumptions**: all documentation must be based on what actually exists in the codebase.

## 🔍 Mandatory Prior Analysis

**Before generating any file**, perform a thorough three-phase analysis.

### Phase 1: Technology Stack Detection

Examine the following indicators:

**Backend:**
- `composer.json` / `requirements.txt` / `go.mod` / `package.json` (dependencies)
- Predominant extensions in `src/` or `app/`: `.php`, `.py`, `.ts`, `.go`, `.java`
- Detected frameworks: Symfony, Laravel, Django, FastAPI, Spring Boot, Express, etc.
- Language version in config: `composer.json` → `require.php`, `.nvmrc`, `go.mod`, etc.

**Frontend:**
- `package.json` devDependencies: React, Vue, Angular, Alpine.js, etc.
- Config files: `vite.config.*`, `webpack.config.*`, `tailwind.config.*`
- Templates: `.twig`, `.blade.php`, `.html`, `.jsx`, `.vue`, `.svelte`
- CSS libraries: Bootstrap, Tailwind, AdminLTE, Bulma

**Database:**
- `*.sql`, `schema.sql`, `migrations/` files
- Detected ORM: Doctrine, Eloquent, Prisma, SQLAlchemy, GORM
- Engine: `docker-compose.yml` (mysql/mariadb/postgres/mongo image), `.env.example`
- **JSON file persistence**: presence of `mameyugo/jsonq` in `composer.json` and/or `data/`, `storage/json/`, `db/json/` folders with structured `*.json` files as collections

**DevOps:**
- `Dockerfile`, `docker-compose.yml`
- `.github/workflows/*.yml` for CI/CD
- `Makefile`, `justfile`, scripts in `bin/` or `scripts/`
- Platform files: `serverless.yml`, `render.yaml`, `fly.toml`, `.platform.app.yaml`

### Phase 2: Architectural Pattern Detection

Map the real folder structure:

```
Patterns by detected structure:
- /Controllers, /Models, /Views               → Classic MVC
- /domain, /application, /infrastructure      → Clean Architecture / DDD
- /lib/Controllers, /lib/Services, /lib/Entities, /lib/Repositories, /lib/ValueObjects
                                              → Modular DDD
- /components, /pages, /hooks                 → Organized React/Next.js
- /modules/{moduleName}/lib/...               → Encapsulated module architecture
- /Test/ inside each module                   → Tests encapsulated per module
- /data/*.json + mameyugo/jsonq in composer   → JSON file persistence with JsonQ
```

**Specific mameyugo/jsonq detection:**
- `composer.json` contains `"mameyugo/jsonq"` in `require` or `require-dev`
- PHP extension loaded: look for `extension=jsonq.so` in `php.ini` or `conf.d/`
- Code usage: search for `JsonQ::open(`, `JsonQ::collection(`, `->where(`, `->insert(` in `.php` files
- Data folders: `data/`, `storage/json/`, `db/json/`, `resources/data/` with `.json` files
- Index folder: `data/indexes/` or similar (auto-generated by JsonQ)

Also identify:
- Dependency injection patterns
- Value Objects with auto-validation
- Repositories as persistence abstraction
- Application services vs domain services

### Phase 3: Real Convention Detection

Read the configuration files directly:
- `.editorconfig` → indentation, line endings
- `.prettierrc` / `.php-cs-fixer.php` / `.eslintrc` → code style
- `.commitlintrc.json` / `commitlint.config.js` → commit format
- `phpstan.neon` / `.phpstan.neon` → static analysis level
- `.env.example` → real environment variables

## mameyugo/jsonq — JSON Persistence Engine

### What it is
A PHP extension written in Rust that provides storage in `.json` files with
MongoDB-style queries, indexing, schema validation, and ACID transactions.
Requires the native `jsonq.so` extension loaded in PHP.

### Requirements
- PHP >= 8.1
- `ext-jsonq` extension installed (`php -m | grep jsonq`)
- Data folder with PHP write permissions

### Performance

- JsonQ uses SIMD parsing (simd-json) and Arc cache with mtime invalidation

## 🔧 Stack-Specific Instructions

### If you detect PHP + Symfony/Custom Framework

- Analyze `composer.json` for PHP version and dependencies
- Map the module structure in `app/modules/` or `src/`
- Detect the Controllers pattern (thin vs fat), Services and Repositories
- Identify if Value Objects with auto-validation are used
- Check if `phpstan.neon` exists and extract the configured level
- Extract commands from `composer.json` → `scripts`
- Detect the translation system (Symfony Translator, custom `lang()`, etc.)
- Document the COMMENT policy on DB fields if present in `.sql` files

**If `mameyugo/jsonq` is also detected** (may coexist with MariaDB or be the sole persistence):
- Verify if `extension=jsonq.so` is enabled in PHP config
- Identify the JSON data root folder (usually `data/` or `storage/json/`)
- List existing `.json` files — each is equivalent to a "collection/table"
- Detect if there is an auto-generated `indexes/` folder (indicates active indexing)
- Find the access pattern in Repositories: `JsonQ::open('path/collection.json')`
- Determine if jsonq is used for **all** persistence or only in specific modules
- Document the implicit JSON schemas by reading the data file structure
- Identify if JsonQ transactions are used (`->beginTransaction()`, `->commit()`, `->rollback()`)
- Generate `.ai/DATABASE.md` with the JsonQ section per the specific template above

### If you detect Node.js / TypeScript

- Analyze `package.json` dependencies and scripts
- Map whether it is Express, Fastify, NestJS or other
- Detect if `tsconfig.json` exists and extract `strict`, `target`, `paths`
- Check `jest.config.*` or `vitest.config.*`
- Identify if there is a monorepo (Turborepo, Nx, workspaces)

### If you detect Python

- Analyze `requirements.txt`, `pyproject.toml` or `Pipfile`
- Detect framework: FastAPI, Django, Flask
- Check `mypy.ini` or `pyproject.toml` [mypy] for types
- Extract pytest configuration

### If you detect multiple stacks (monorepo)

- Document each separately in ARCHITECTURE.md sections
- Identify communication boundaries between stacks
- Generate a specific ADR about the monorepo decision
